MIRA — AUV Perception Stack

An underwater vehicle that sees in the dark.

fig. — MIRA, Team Dreadnought's competition AUV

The full computer-vision perception stack for Team Dreadnought's autonomous underwater vehicle — custom object detection, sonar preprocessing, and real-time inference driving navigation and task execution at international competitions.

role
Perception lead · Team Dreadnought
stack
YOLO · OpenCV · ROS · Python · Roboflow
result
deployed at international AUV competitions
view source ↗

architecture

Camera and sonar streams are preprocessed on board, fed to a custom-trained YOLO detector, and published into the vehicle's ROS graph — where navigation and control subsystems consume detections in real time.

challenge

Underwater perception is hostile: light distortion, turbidity, and colour loss break models trained on clean data — and everything must run within the latency budget of a live competition task, on the vehicle's own hardware.

engineering decisions

A domain-specific dataset annotated and versioned in Roboflow, with augmentation tuned for underwater distortion; sonar preprocessing to clean acoustic returns; and a modular ROS integration so perception evolves without touching control.

results

A perception stack that runs in real time on board and has been exercised under real competition conditions at international AUV challenges.

PRJ 02 · complete

3-DOF Gesture-Controlled Arm

A robotic arm that mirrors the human hand.

demo — the arm mirroring the human hand

Real-time hand tracking driving a 3-DOF manipulator end-to-end — from camera to MediaPipe landmark detection to joint-angle mapping to servo motion — validated on physical hardware at Coratia Technologies.

role
AI & Robotics Intern · Coratia Technologies
stack
MediaPipe · OpenCV · PyFirmata · Arduino
result
validated on physical hardware
view source ↗

architecture

An OpenCV capture loop feeds MediaPipe, which resolves 21 keypoints per hand at camera frame rate. Landmark geometry is mapped to three joint angles, sent over a PyFirmata serial bridge to Arduino-controlled servos.

challenge

Raw hand landmarks are noisy, and servos amplify every jitter into visible shaking. The arm had to feel like a mirror of the hand — smooth, immediate, and stable on real hardware, not just in a demo window.

engineering decisions

Smoothing between landmark detection and actuation to keep physical motion stable; a clean joint-angle mapping for all three degrees of freedom; and dual-hand tracking so each hand can own a different control axis.

results

An end-to-end gesture control system validated on a physical 3-DOF arm during the internship at Coratia Technologies.

PRJ 03 · complete

Custom Object Detection Pipeline

Teaching a model to find what matters underwater.

fig. — feature field & confirmed detection

A complete train-to-deploy pipeline: domain dataset annotated in Roboflow, custom YOLO training with augmentation tuned for underwater distortion, and inference optimised for real-time performance on edge hardware.

role
end-to-end vision pipeline
stack
YOLO · Roboflow · PyTorch · OpenCV
result
real-time on edge, shipped into MIRA
view source ↗

architecture

Dataset annotation and versioning in Roboflow with controlled splits; custom YOLO training in PyTorch; then an optimisation pass balancing model size, input resolution, and pipeline latency for the deployment target.

challenge

Domain shift: pretrained detectors collapse on underwater imagery, and the deployment hardware is a constrained edge device — accuracy had to survive both the water and the compute budget.

engineering decisions

Building a domain-specific dataset rather than fine-tuning on generic data; versioned augmentation policies per experiment; and treating real-time inference as a design constraint from day one, not an afterthought.

results

Real-time detection on edge hardware, shipped into the MIRA competition perception stack and exercised under live competition conditions.

PRJ 04 · complete

MPi — Market Intelligence

Reading a nation's payment flows before they happen.

fig. — observed series → forecast horizon

An end-to-end pipeline over RBI Payment Systems data — automated Playwright ingestion, Excel-to-Parquet processing, and Prophet time-series forecasting, written up in a production architecture design doc.

role
data engineering & forecasting
stack
Prophet · Playwright · Pandas · Parquet
result
2.1% MAPE on credit-card volumes
view source ↗

architecture

Playwright automation scrapes the RBI DBIE portal; raw Excel releases are parsed and normalised into columnar Parquet; Prophet models forecast credit and debit card volumes from the cleaned series.

challenge

The source is a dynamic government portal publishing irregular Excel files — the pipeline had to survive both the portal's structure and the data's inconsistency before any modelling could begin.

engineering decisions

Browser automation over brittle URL guessing; Parquet for fast, typed analysis; Prophet for interpretable seasonal decomposition; and a design doc describing how the prototype scales into a scheduled, monitored production system.

results

The credit-card volume model reached 2.1% MAPE, with the full path from raw portal to forecast automated end-to-end.

PRJ 05 · active

Deep Learning Journey

Deep learning, rebuilt by hand from zero.

fig. — a loss curve settling into its minimum

A structured, ground-up curriculum built in public — neural networks and backpropagation from scratch in NumPy, then PyTorch training loops, CNNs, and transfer learning. Every concept proven in runnable code.

role
self-directed curriculum
stack
NumPy · PyTorch · Jupyter
result
46 notebooks, built in public
view source ↗

architecture

A deliberate progression: forward passes, backpropagation, and gradient descent by hand in NumPy first; then PyTorch tensors, autograd, and training loops; then CNNs and transfer learning applied to vision problems.

challenge

Tutorials create the illusion of understanding. The goal was the real thing — being able to rebuild every layer of the stack from an empty file, without reference.

engineering decisions

From-scratch implementations before frameworks; one runnable notebook per concept; and building in public so the record of progress is honest and inspectable.

results

46 commits of working notebooks covering the path from raw NumPy to transfer learning — the foundation under the vision systems in this archive.

PRJ 06 · complete

Agri-Mitra — Smart India Hackathon

Field intelligence for the people who feed us.

fig. — growth from a single field line

An agricultural intelligence platform — crop recommendations, disease detection guidance, and market price data for farmers. Contributed the computer-vision and backend data pipeline components.

role
CV & data pipeline · team build
stack
JavaScript · Computer Vision · Web
result
built for Smart India Hackathon
view source ↗

architecture

A JavaScript web application backed by a data pipeline serving market prices and crop recommendation inputs, with computer-vision components providing disease detection guidance.

challenge

Hackathon constraints: a multidisciplinary team, a hard deadline, and users — farmers — for whom the interface had to be simple even though the system behind it was not.

engineering decisions

A clean split between the vision components and the data pipeline so the team could build in parallel; recommendations delivered as plain guidance rather than raw model output.

results

A working platform delivered for the Smart India Hackathon, with the computer-vision and backend data components contributed to the team build.
